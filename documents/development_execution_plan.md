# City Flaneur Agent 开发执行计划（WBS + 排期 + 分工）

## 1. 目标与交付

### 1.1 目标
在课程作业周期内，完成一个可演示的 LLM Agent 系统，满足以下硬性要求：
1. 展示规划与思考能力。
2. 至少接入 2 个外部工具并用于决策。
3. 具备异常处理与自我修正能力。
4. 可视化内部状态流转。

### 1.2 最终交付物
1. 可运行系统（Web 优先）。
2. 需求文档 + 执行计划文档。
3. 演示脚本与测试用例集。
4. 录屏或现场演示流程（含突发场景）。

## 2. 推荐技术栈（执行版）

### 2.1 主推荐方案
1. 前端：Next.js + TypeScript + React。
2. Agent：LangGraph + LangChain。
3. 模型：OpenAI Responses API。
4. 数据层：Prisma + SQLite（开发），可迁移 PostgreSQL。
5. 可视化：状态时间线组件 + 结构化日志。

### 2.2 备选方案
1. Python 后端方案：FastAPI + LangGraph(Python) + Next.js 前端。
2. 纯 TypeScript 全栈：Next.js Route Handler + LangChain JS。

### 2.3 选型结论
若团队偏向快速交付和统一开发体验，优先采用“纯 TypeScript 全栈方案”。

## 3. WBS 任务分解（按模块）

### 3.1 模块 A：Agent 核心（P0）
1. A1 目标澄清器（Clarifier）
2. A2 初始计划生成器（Planner）
3. A3 执行与观测循环（Act/Observe）
4. A4 反思与重规划器（Reflect/Replan）
5. A5 推理策略路由（ReAct -> Plan-and-Execute）

完成定义（DoD）：
1. 支持模糊输入到结构化目标。
2. 至少 3 步计划输出。
3. 可在异常时触发重规划。

### 3.2 模块 B：工具层（P0）
1. B1 天气工具封装（统一输入/输出 schema）
2. B2 地图/POI 工具封装（营业状态、通行时间）
3. B3 偏好记忆读写工具
4. B4 工具错误处理与超时重试策略

完成定义（DoD）：
1. 工具成功、超时、空结果三类路径可测。
2. 工具结果能影响计划而非只展示。

### 3.3 模块 C：前端与可视化（P0）
1. C1 对话输入与计划输出 UI
2. C2 状态流转时间线（Thought/Action/Observation）
3. C3 重规划前后 diff 展示
4. C4 演示模式（可回放关键日志）

完成定义（DoD）：
1. 能看到完整状态链路。
2. 能定位某次重规划的触发原因。

### 3.4 模块 D：数据与存储（P0/P1）
1. D1 数据模型定义（UserProfile/PlanStep/Observation）
2. D2 偏好更新策略（喜欢/不喜欢权重）
3. D3 会话日志持久化
4. D4 基础统计（成功率、重规划次数）

### 3.5 模块 E：测试与演示（P0）
1. E1 单元测试（工具层、异常处理）
2. E2 集成测试（端到端核心链路）
3. E3 演示脚本（正常场景 + 突发场景）
4. E4 验收清单打勾（映射课程要求）

## 4. Backlog（优先级）

### 4.1 P0（必须完成）
1. 模糊目标输入 -> 结构化任务。
2. 双工具接入（天气 + 地图/POI）。
3. 偏好记忆读写。
4. 异常检测与自动重规划。
5. 状态流转可视化。
6. 至少 1 个突发场景演示成功。

### 4.2 P1（强烈建议）
1. 多推理策略切换（ReAct + Plan-and-Execute）。
2. 双方案比较（舒适优先/效率优先）。
3. 预算与体力约束建模。
4. 反馈闭环学习。

### 4.3 P2（有余力再做）
1. 结果导出（文本卡片）。
2. 多语言输出。
3. 可观测性增强（指标看板）。

## 5. 4 周排期（建议）

### Week 1：架构搭建与最小链路
1. 搭项目骨架（前端、接口、Agent 状态机）。
2. 完成 Clarifier + Planner 基础版本。
3. 接入天气工具（可 mock）。
4. 输出第一个可跑通 Demo（无重规划）。

里程碑验收：
1. 可输入需求并返回基础行程建议。

### Week 2：工具完善与自我修正
1. 接入地图/POI 工具并与规划联动。
2. 实现异常分类（闭馆、超时、拥堵）。
3. 完成 Reflect/Replan 节点。
4. 增加重试/降级策略。

里程碑验收：
1. 出现闭馆场景可自动重规划。

### Week 3：可视化与记忆系统
1. 完成状态时间线 UI。
2. 完成偏好记忆读写与反馈更新。
3. 展示重规划前后差异。
4. 引入第二种推理策略（加分项）。

里程碑验收：
1. 完整展示 Thought -> Action -> Observation -> Replan。

### Week 4：测试、压测、演示打磨
1. 补齐单元与集成测试。
2. 打磨演示脚本与容错预案。
3. 优化响应速度与稳定性。
4. 完成最终文档与答辩材料。

里程碑验收：
1. P0 全部通过，P1 至少完成 2 项。

## 7. Issue 模板（可直接开任务）

### 7.1 功能任务模板
1. 任务标题：`[P0][模块X] xxx功能`
2. 输入：用户故事 + 依赖项。
3. 输出：接口/页面/日志变更。
4. 验收：Given-When-Then 三段式。
5. 风险：失败模式与兜底方案。

### 7.2 缺陷任务模板
1. 现象：异常触发路径。
2. 复现步骤：最小复现输入。
3. 预期行为：系统应如何自修正。
4. 修复标准：测试新增与回归通过。

## 8. 验收用例（最小集）

### 8.1 正常流程
1. 输入：上海艺术感下午 + 不想太累。
2. 预期：输出含 3+ 步行程与理由。

### 8.2 突发闭馆
1. 输入：第一站设为某美术馆；工具返回闭馆。
2. 预期：触发重规划，替换为同类候选点。

### 8.3 天气突变
1. 输入：原计划包含户外点；天气 API 返回降雨。
2. 预期：优先切换室内方案并调整交通。

### 8.4 记忆生效
1. 输入：用户反馈“更喜欢古建筑”。
2. 预期：下一轮计划古建筑权重明显提升。

### 8.5 工具失败降级
1. 输入：地图 API 超时。
2. 预期：触发重试；失败后给出保守建议而不崩溃。

## 9. 演示脚本（8~10 分钟）

1. 第 1 分钟：项目目标与作业要求映射。
2. 第 2~3 分钟：正常规划流程演示。
3. 第 4~6 分钟：闭馆/天气突发触发自我修正。
4. 第 7 分钟：展示状态流转时间线与关键日志。
5. 第 8 分钟：展示偏好记忆生效。
6. 第 9~10 分钟：总结亮点（多策略推理、鲁棒性）。
