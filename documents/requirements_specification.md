# “随机应变”的城市深度游助手（City Flaneur Agent）需求规格说明书

## 1. 文档目的与范围

### 1.1 文档目的
本文件用于将 `requirements_initial.md` 中的初始想法与课程作业要求转化为可执行、可验收、可迭代的完整需求规格说明，作为后续设计、开发、测试与演示的统一依据。

### 1.2 项目目标
构建一个面向“城市轻量深度游”场景的 LLM Agent 系统，能够在目标模糊、环境动态变化（天气、闭馆、排队、交通拥堵）情况下，持续规划、执行、观察、修正，并以可视化方式展示内部状态流转。

### 1.3 与课程作业要求映射
1. 具备规划与思考能力：通过多阶段任务规划（目标澄清 -> 候选方案 -> 执行策略 -> 实时修正）实现。
2. 具备至少 2 个外部工具：天气/地图等实时工具 + 用户偏好记忆工具。
3. 具备自我修正能力：异常捕获、失败分类、自动重规划。
4. 可视化内部状态流转：在 Web/CLI 中展示 Thought -> Action -> Observation 轨迹。
5. 加分项（可选）：组合 ReAct + Plan-and-Execute + Reflexion 等策略。

## 2. 项目背景与问题定义

### 2.1 业务痛点
传统旅游攻略静态、固定、低个性化；现实出行中常见临时变化导致“攻略失效”，用户需要高频手动重查并重新决策，认知负担高。

### 2.2 核心问题
在“目标模糊 + 环境不确定 + 偏好动态变化”的条件下，如何让 Agent 提供持续、可解释、可自我修复的行程建议。

### 2.3 目标用户
1. 轻度旅行用户：希望“有氛围、不折腾、不做复杂决策”。
2. 城市漫游用户：偏好临场探索而非打卡。
3. 课程演示评审方：关注 Agent 智能性、可观测性、工程完整性。

## 3. 术语与定义

1. Agent：可进行推理、调用工具、执行动作并根据反馈修正计划的智能体。
2. 工具（Tool）：Agent 可调用的外部能力接口，如天气查询、地点详情、记忆库读写。
3. 自我修正（Self-correction）：执行失败或上下文变化后，系统主动回退、重估并产出新方案。
4. 状态流转可视化：展示推理过程与执行过程关键节点（计划、动作、观测、反思、重试）。

## 4. 需求边界

### 4.1 In Scope（本期范围）
1. 城市内半日/一日行程的生成与动态重规划。
2. 至少 2 个外部工具接入并联动决策。
3. 个人偏好记忆与会话内上下文管理。
4. 执行日志和状态可视化（Web 优先，CLI 备选）。

### 4.2 Out of Scope（本期不做）
1. 在线支付、票务下单、酒店预订闭环。
2. 跨城市多日复杂行程优化（可作为后续扩展）。
3. 高精度地图路径引擎自研（依赖第三方地图服务）。

## 5. 角色与使用场景

### 5.1 角色定义
1. 游客用户（Primary User）：输入偏好、接收建议、反馈体验。
2. 系统管理员/开发者：维护提示词、工具配置、日志监控。
3. 课程评审者：检查作业需求覆盖与演示效果。

### 5.2 关键场景
1. 模糊目标启动：
   - 输入“我想在上海度过有艺术感的下午，不想太累”。
   - Agent 追问约束（时间/预算/步行容忍度），输出候选方案。
2. 执行中突发变化：
   - 美术馆闭馆或排队过长。
   - Agent 检测异常后替换为附近替代点，并重排交通顺序。
3. 偏好变化：
   - 用户临时表示“更想看古建筑”。
   - Agent 更新偏好权重并重新规划剩余行程。

### 6.2 详细功能需求

#### FR-01 目标澄清与任务建模（P0）
1. 系统应支持接收模糊目标输入。
2. 系统应通过 1~3 个关键追问补全约束（时间、体力、预算、交通偏好）。
3. 系统应形成结构化任务对象（目标、约束、偏好、不可接受条件）。

验收标准：
1. 对任意模糊输入，均可输出结构化目标模型。
2. 若缺失关键约束，必须触发追问而不是直接规划。

#### FR-02 多步计划生成与执行（P0）
1. 系统应生成包含时间段、地点、交通方式、预估停留时长的多步计划。
2. 每一步应可执行并可被后续步骤引用。
3. 执行过程中应记录动作与观测结果。

验收标准：
1. 输出计划包含 >= 3 个可执行步骤。
2. 每步均有可解释字段（为什么选它）。

#### FR-03 天气/地图实时工具调用（P0）
1. 系统应接入天气查询工具。
2. 系统应接入地图地点详情/通行时间工具。
3. 工具结果应参与决策而非仅展示。

验收标准：
1. 天气异常（降雨、高温）可触发方案调整。
2. 目标地点闭馆/拥挤可触发替代点推荐。

#### FR-04 偏好记忆读写工具（P0）
1. 系统应支持读取用户历史偏好（例如艺术、建筑、步行上限）。
2. 系统应支持根据用户反馈更新偏好权重。
3. 新计划应显式反映偏好变化。

验收标准：
1. 连续两轮会话中，第二轮计划体现第一轮反馈。
2. 记忆写入失败时，系统应降级处理并提示。

#### FR-05 异常检测与自我修正（P0）
1. 系统应识别工具调用失败、目标不可达、条件突变等异常。
2. 出现异常时应执行“回退 -> 诊断 -> 重规划”流程。
3. 应限制循环重试次数并给出用户可选方案。

验收标准：
1. 能稳定处理至少 3 类异常（闭馆、天气突变、接口失败）。
2. 失败后不崩溃，输出可执行替代方案或退出建议。

#### FR-06 状态流转可视化（P0）
1. 系统应展示 Thought / Plan / Action / Observation / Reflection 状态节点。
2. 应支持查看单次任务的时间线与关键决策点。
3. 应标记“触发修正”的原因和前后计划差异。

验收标准：
1. 演示中可清晰观察至少一次自我修正全流程。
2. 关键工具调用参数与摘要结果可追溯。

#### FR-07 多推理框架编排（P1）
1. 默认采用 ReAct（快速闭环）。
2. 复杂场景触发 Plan-and-Execute（先全局后局部）。
3. 失败后引入 Reflexion 进行经验反思与策略修订。

验收标准：
1. 同一任务可展示至少两种推理策略切换。

#### FR-08 多方案比较与解释（P1）
1. 系统应输出至少 2 套备选行程（舒适优先/效率优先）。
2. 每套方案应给出差异说明与适用条件。

#### FR-09 预算/体能约束建模（P1）
1. 支持预算区间、最大步行距离、最大连续活动时长约束。
2. 约束冲突时需提示并给出妥协建议。

#### FR-10 用户反馈闭环学习（P1）
1. 行程结束后收集满意度反馈。
2. 将“喜欢/不喜欢”写入记忆库并影响后续推荐。

#### FR-11 结果导出（P2）
1. 支持导出为文本摘要或可分享卡片。

#### FR-12 多语言输出（P2）
1. 支持中文/英文输出切换。

## 7. 非功能需求

### 7.1 性能
1. 普通请求首版计划生成时间 <= 8 秒（无重规划）。
2. 触发一次重规划的额外耗时 <= 6 秒。

### 7.2 可靠性
1. 工具失败应可重试并降级。
2. 系统整体可用性目标 >= 95%（课程演示阶段）。

### 7.3 可解释性
1. 对关键选择提供简短理由。
2. 记录每次重规划的触发依据。

### 7.4 安全与隐私
1. 不存储敏感个人身份信息。
2. API Key 使用环境变量管理。
3. 日志脱敏（位置可模糊化）。

### 7.5 可维护性
1. 推理策略、工具层、UI 展示解耦。
2. 核心模块具备基础单元测试与集成测试。

## 8. 数据与接口需求

### 8.1 核心数据实体
1. UserProfile：偏好标签、预算、体力参数、历史反馈。
2. SessionContext：当前城市、时间窗口、任务状态。
3. PlanStep：步骤类型、地点、开始时间、时长、解释。
4. ToolObservation：工具名称、输入摘要、输出摘要、状态码。
5. CorrectionRecord：异常类型、修正策略、前后差异。

### 8.2 外部接口（最小集）
1. Weather API：天气状况、降雨概率、温度区间。
2. Maps/POI API：地点营业状态、拥挤度（若可用）、路程时间。

### 8.3 内部工具接口
1. `get_weather(city, time_window)`
2. `search_poi(query, location, constraints)`
3. `read_user_memory(user_id)`
4. `write_user_memory(user_id, feedback)`

## 9. 状态机与自我修正规则

### 9.1 状态机
1. `INIT`：接收输入。
2. `CLARIFY`：补齐约束。
3. `PLAN`：生成候选计划。
4. `ACT`：执行步骤并调用工具。
5. `OBSERVE`：接收执行反馈。
6. `REFLECT`：评估是否偏离目标。
7. `REPLAN`：局部或全局重规划。
8. `DONE`：输出最终建议与复盘。

### 9.2 自我修正触发条件
1. 地点不可用（闭馆/超长排队/不可达）。
2. 外部条件变化（天气恶化、交通突发拥堵）。
3. 用户偏好突变（临时更换兴趣方向）。
4. 工具调用异常（超时、空结果、冲突结果）。

### 9.3 修正策略
1. 局部替换优先：只替换当前失败节点及相邻步骤。
2. 全局重排兜底：若多点冲突则重建后续计划。
3. 失败退出策略：给出“人工选择”分支，避免死循环。

## 10. 验收标准（按里程碑）

### 10.1 M1：最小可运行版本（必须）
1. 支持模糊目标输入与结构化任务建模。
2. 接入 2 个外部工具。
3. 具备至少 1 条可视化状态时间线。

### 10.2 M2：核心智能能力（必须）
1. 可稳定触发并完成自我修正。
2. 支持偏好记忆读写。
3. 演示至少 1 个真实“突发场景”闭环。

### 10.3 M3：增强与加分（建议）
1. 展示多推理框架组合。
2. 提供多方案比较和差异解释。

## 11. 风险与缓解

1. 外部 API 不稳定：
   - 缓解：增加缓存、重试、降级文案与替代数据源。
2. 规划质量波动：
   - 缓解：引入计划评分器（约束满足度 + 用户偏好匹配度）。
3. 状态可视化信息过载：
   - 缓解：默认摘要视图 + 展开详细日志。
4. 演示环境网络抖动：
   - 缓解：准备离线回放数据和预录日志样例。

## 12. 推荐技术框架与选型建议

### 12.1 总体推荐（课程作业最优平衡）
1. 前端：Next.js（App Router）+ React + TypeScript。
2. Agent 编排：LangGraph（状态机与可视化友好）+ LangChain（工具抽象）。
3. 模型接入：OpenAI Responses API（或兼容模型供应商）。
4. 后端接口：Next.js Route Handlers（轻量一体化）或 FastAPI（若需独立服务）。
5. 记忆存储：
   - 开发阶段：SQLite + Prisma。
   - 可扩展阶段：PostgreSQL + pgvector（偏好向量检索）。
6. 可视化与观测：
   - 前端自建状态时间线 + 结构化日志。
   - 可选接入 LangSmith / OpenTelemetry 做链路追踪。

### 12.2 为什么推荐 LangGraph
1. 天然适配“有状态、多节点、可中断可恢复”的 Agent 工作流。
2. 非常适合实现 Self-correction（节点失败后转入 Replan 节点）。
3. 易于把内部状态映射到 UI，满足课程对“可视化”的硬性要求。

### 12.3 推理框架组合建议
1. 主流程：ReAct（快速工具调用与反馈闭环）。
2. 复杂任务：Plan-and-Execute（先分解再执行）。
3. 失败复盘：Reflexion（记录失败原因并调整策略）。

### 12.4 备选方案
1. 若团队更熟悉 Python：前端 Next.js + 后端 FastAPI + LangGraph（Python）。
2. 若追求全栈 TypeScript：前后端统一 TS，Agent 层使用 JS 生态 LangChain/LangGraph。
